## [v0.7.1] - 2026-05-15

### 新增功能 (Added)
* 新增 text follow 的三层结构化契约 `token -> character -> byte`，在保留现有分 token 规则的前提下，为每个 token 额外发布 character entry 和带 character metadata 的 raw byte entry。这样 Android 不再需要自己从 UTF-8 文本猜字符边界，后续可以直接用 libs 提供的数据做 token 高亮、character 高亮和 Binary/Hex 跟随。
* 新增 character 语义分类 `Visible / Space / Newline / SeparatorOther`，并把字符文本通过 `offset + size` 形式暴露到 `bag_api`。这样 Android 可以直接知道当前跟随的是可见字符、空格还是换行，更方便在 token 卡片里表达“空白但可高亮”的空格位置，以及后续换行相关显示。

### 技术改进/重构 (Changed/Refactor)
* 重构 `audio_core` 的 text follow 构建逻辑，使 tokenization 继续由 libs 统一决定，同时在同一条解析链路里补齐 character 与 byte 的映射关系，包括 `character_index_within_token`、`byte_index_within_character`、`character_byte_count`、`is_character_start/end`。这为 Android 渲染 CJK 文本时每个字符内部的 `3 x 8bits` 分组、字符边界 divider 和 mixed UTF-8 token 的 byte 对齐提供了直接依据。
* 重构 `bag_api` follow marshalling、结果布局和同步测试，让新的 character layer、separator character 文本缓冲区和 byte metadata 可以稳定跨 ABI 输出，并与 `audio_core` 的实际 follow 数据保持一致。这样 Android JNI 可以直接消费 libs 的 source-of-truth 数据，不再依赖前端 merge/split token 的兼容逻辑。
* 补充 text follow contract 和 `libs/AGENTS.md` 指引，明确 text follow / token-character-byte 改动先看 `docs/architecture/text-follow-contract.md`。这样新的 agent 或后续维护者能先理解“token 规则、separator 语义、byte 边界”这套正式契约，再去改实现。

### 修复 (Fixed)
* 修复 text follow 只提供 token 和裸 byte、导致前端需要自己推断 trailing space/newline 与字符边界的问题。现在 libs 会把这些 separator 作为正式 character 语义发布，Android 可以在空格播放到时继续保持 token 跟随和 Binary/Hex 跟随一致。
* 修复 CJK 和其他多字节字符在 follow 数据里缺少 character-byte 对齐信息的问题。现在每个 byte 都会明确标注自己属于哪个字符及字符内位置，Android 可以稳定画出 CJK 文本的 byte divider，而不再把多字节字符当成不可拆分的黑盒 byte 串。
