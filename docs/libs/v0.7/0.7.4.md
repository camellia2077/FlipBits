## [v0.7.4] - 2026-05-24

### 新增功能 (Added)
* 新增 encode operation 生成进度公共契约，把 Web 和 Android 的进度条数据统一收回到 `libs` 提供的 source-of-truth：`bag_encode_operation_progress` 暴露 state、phase、overall/phase progress、work units、payload/segment 信息，`bag_encode_operation_work_plan` 暴露生成前的静态工作计划，消费者不再需要各自推导 phase、percent 或总工作量。
* 新增 operation pump / poll / work-plan / terminal-result 的 C API 与 `audio_core` 对应模型，让 `mini / flash / pro / ultra` 都能用同一套 `bag_create_encode_operation`、`bag_pump_encode_operation`、`bag_poll_encode_operation` 和 `bag_take_encode_operation_result` 生命周期对外发布生成状态。

### 技术改进/重构 (Changed/Refactor)
* 重构生成进度架构，动机是此前为 Web 增加生成中进度显示后，`libs` 进度接口变更让 Android 侧出现生成卡在 `0%` 的回归；随后虽然恢复了 Android 生成，但又把原本十几个字符不到一秒的生成拉长到十几秒。现在进度显示、native work 推进、终态结果和 follow hydration 被拆成明确边界，避免 Web 与 Android 为同一段生成过程各自消费不同语义。
* 重构 `audio_core` 的 encode operation 执行模型，把 `mini / flash / pro / ultra` 渲染拆成可 pump 的阶段化工作：preparing input、rendering PCM、postprocessing、finalizing。pump budget 只限制 native 单次推进量和 wall time，UI 刷新节奏由 Android/Web 自己节流，不再通过放慢 native pump 来制造可见进度。
* 调整 operation 结果层边界：`bag_take_encode_operation_result` 只承担终态 PCM 结果；raw/follow/timeline 展示数据继续通过 `bag_build_encode_follow_data` 单独 hydration。`bag_encode_result`、follow array 和 text follow buffer 的注释也明确了 `*_count` / `*_size` 可以作为 sizing probe，不能在 status 不可读时被当成真实可读 buffer。
* 补充 `libs/AGENTS.md`、`libs/README.md` 和架构文档里的生成进度入口说明，明确当前权威契约是 encode operation；后续改 snapshot、work-plan、pump 或 Android/Web 生成显示时必须同步检查 consumer bridge 和测试。

### 修复 (Fixed)
* 修复 Android 在新 `libs` 进度契约下可能一直显示 `0%`、无法按正常进度进入生成完成状态的问题。Android/Web 现在都以 operation snapshot 和 work-plan 为生成进度事实来源，Android 侧不再做本地二次推导。
* 修复 Android 生成速度显著回退的问题：native operation 可以连续 pump 工作，UI progress emission 单独节流；同时修复 Flash postprocess 在每个小 pump 后重复复制完整 PCM 的 O(n^2) 行为，避免 Flash 长文本生成被反复全量拷贝拖慢。
* 修复 operation terminal result 与 follow hydration 语义混在一起导致的 ABI 使用风险。现在调用方可以只取 PCM 完成生成，再按需要单独请求 follow/raw/timeline 数据，避免因为未提供 caller-owned follow buffer 就把 sizing count 误当成可读内容。
