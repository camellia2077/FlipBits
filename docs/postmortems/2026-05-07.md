# Android Flash Visual Animation Stutter Postmortem

## 1. Problem

Android Flash Visual 在长文本、长音频场景下出现了明显的动画不流畅问题。用户肉眼看到的现象包括：

- Visual 播放时帧率感知变低。
- 可视图向左平移时会周期性跳动。
- 部分阶段出现闪烁或瞬移。
- 长音频比短音频更容易复现。

这个问题只存在 Android 可视化路径。CLI 没有波形图、Flash Visual、Canvas 动画或 Lyrics 播放跟随 UI，因此不应该先把它当成 CLI 或 libs 的通用问题处理。

后续 debug 数据证明，问题也不是简单的 FPS 低。加入 debug overlay 后，Visual 播放时 `draw/s` 基本能到约 100 FPS，`drawAvgMs` 也很低。也就是说，人眼感觉“卡”并不等价于“FPS 太低”。这次真正的问题是视觉时间轴发生跳变：raw playback position 更新是粗粒度的，而 visual smoother 又被 raw anchor 周期性重启，导致 Canvas 虽然画得很快，但 viewport 会间歇性跳。

## 2. Initial Approach

最开始的处理方式是错误的工程方法。看到长音频卡顿后，没有先读清关键代码路径，也没有先要求设备侧数据，而是把“切换更强的 agent / 更高 reasoning 等级”当成主要解决手段：ChatGPT 5.4 high 没有解决，就切到 ChatGPT 5.5 high；ChatGPT 5.5 high 仍然没有解决，就切到 ChatGPT 5.5 xhigh。开发者不应该抱着“多充点钱买余额、切换更强模型、切换更高思考等级”就能解决难问题的心态。这个做法本质上是在偷懒，不是诊断。

开发者当时主要提供的是主观感受：长音频播放 Visual 时“感觉卡”“感觉帧率低”。后续加入 debug 性能可视化以后，数据证明这个判断是错的：Visual 的 FPS 基本能稳定到 100 左右，`drawAvgMs` 也很低。问题不是低 FPS，而是 visual viewport 因 smoother reset 周期性跳变。开发者把一个主观判断当成事实输入给 agent，导致 agent 也更容易沿着错误方向找原因。

在这种输入质量下，agent 猜测了很多可能方向：

- 完整 follow data 太大。
- Android 侧 groupBy / filter / index 查找太重。
- LazyColumn 或 Lyrics wrapped line 计算太重。
- Canvas primitive 数太多。
- FlashVisualWindowSource / State 窗口化不足。
- libs/native 应该提供更多 indexed timeline 信息。

这些方向并非完全没有价值，有些确实能降低风险或改善局部性能。但是当时最大的问题是：没有先用设备侧数据确认真正瓶颈，就扩大了修改范围。甚至一度把问题推向 libs/native 数据结构和更大的窗口化重构，但这些改动没有命中根因，部分尝试还让动画节奏更差。

这不是“模型不够聪明”的问题，而是输入和流程不合格。agent 能力强，靠少量代码读取和合理猜测可以修掉很多普通问题；但遇到动画播放不流畅这种需要真实设备反馈的问题，只给主观感受，不给 adb、不加 trace、不提供客观指标，就是让 agent 在黑箱里猜。猜对了是运气，猜错了就会继续扩大改动范围，制造更多代码噪声和长期维护风险。

也不能期待 agent 默认主动做最高成本的诊断路径。出于 token 成本和交互成本，开发者没有明确要求时，agent 通常会先尝试低成本路径：读一部分代码、提出可能原因、做局部修改。它不会在每个 UI 动画问题一开始就展开大量代码输入，也不会默认要求开发者抓 adb。尤其 adb 复现本身并不复杂：开发者只需要生成长文本音频、播放 Visual、开始 logcat 捕获，再把结果提供出来。这个低成本但高价值的信息输入，应该由开发者主动提供或至少主动要求建立。

这次早期处理的明确错误是：开发者试图用“付更多钱买模型余额、换更强模型和更高思考等级来多次尝试”代替“读代码路径、加 debug 指标、抓设备数据”。这是一种偷懒方法，更偏向于祈祷/黑盒赌博的做法，不是可靠的工程方法。

## 3. Engineering Gap

这次问题暴露出几个工程缺口。

第一，文档没有把 Android Visual 的真实数据流写清楚。Flash Visual 到底从哪里进入、吃 raw PCM 还是 follow timeline、哪些数据来自 libs/native、Android 侧哪里做 window/source/state、Canvas 每帧读什么、Lyrics 和 Visual 是否共用 followData，这些都没有一个集中说明。结果排查时容易从大文件里盲搜，也容易把 Android UI 动画问题误判成 libs 层问题。

第二，没有一开始就把系统变成可观测的。人眼只能说“卡”“跳”“闪”，这只能作为现象描述，不能定位层级。动画问题至少要区分：

- FPS 低，画不动。
- Canvas draw 慢。
- Compose 重组过多。
- window build 或数据构建太频繁。
- raw playback position 本身更新粗粒度。
- smoother 重置或 viewport 发生跳变。
- 性能正常，但 bit 信息密度太高，人眼读不住。

第三，debug overlay 一开始只显示 FPS/draw 耗时还不够。后来发现即使 FPS 接近 100，视觉仍然会跳，所以必须继续加入 `rawUpdate/s`、`smoothReset`、`pxStepMax`、`viewportStartStepMaxMs` 等更贴近“视觉运动连续性”的指标。不能主观感觉卡就直接判断是帧率问题。

## 4. Data-Driven Diagnosis

后续修复转向了设备侧数据驱动。Android debug build 增加了 `FlashVisualPerfTrace`，并通过 adb 捕获实际播放数据：

```powershell
adb logcat -c
adb logcat FlashVisualPerf:D *:S > temp\log.txt
python temp\format_flash_visual_perf.py
```

`temp/log.md` 里的关键 summary 显示：

- Avg FPS: `94.14`
- Avg draw ms: `0.35`
- Max draw ms: `2.01`
- Avg visible primitives: `43.41`
- Max px step: `166.43`
- Max anchor jump ms: `185.76`
- Max viewport start step ms: `169.14`
- Total smooth resets: `135`
- Avg window builds/s: `0.88`

这些数据改变了判断方向：

- `draw/s` 高、`drawAvgMs` 低，说明 Canvas throughput 不是主要问题。
- `windowBuild/s` 低，说明窗口构建不是当前主要瓶颈。
- `rawUpdate/s` 约 8-9/s，说明播放器 raw position 本身是粗粒度更新。
- `smoothReset` 每秒约 8-9 次，和 raw update cadence 对齐。
- `pxStepMax` 和 `viewportStartStepMaxMs` 很高，说明不是“画不出来”，而是 visual viewport 被周期性拉回或跳动。

这组指标最终把问题定位到 visual smoother：它用 `LaunchedEffect(anchorSample, ...)` 作为 key，raw anchor 每次变化都会重启协程，并把 visual sample 直接重设为 anchor。Canvas 仍然在高 FPS 绘制，但绘制的 viewport 被周期性重置，所以人眼看到的是跳动和闪烁。

## 5. Fix

最终有效修复是一个窄改动，而不是继续扩大重构范围。

修复方向：

- 不再让 visual smoother 以 `anchorSample` 作为 `LaunchedEffect` key。
- 使用 frame-driven clock 连续推进 smoothed visual sample。
- 通过 updated state 读取最新 raw anchor。
- 只在 smoothed visual sample 和 raw anchor 漂移过大时重新对齐。

这样 raw playback position 仍然可以粗粒度更新，但 Visual Canvas 使用连续的平滑位置移动。修复后，长音频播放终于不再明显卡顿，也不再周期性跳跃。

这次也调整了 debug 指标展示位置和内容：

- debug 性能指标放到歌词/bit 文本上方，避免遮挡 Visual。
- overlay 不只显示 FPS，还显示 raw update、smooth step、visual error、px step、smoother reset 等指标。
- `temp/format_flash_visual_perf.py` 用于把 adb log 转成可读 Markdown。

这次修复后补了文档和索引，避免未来再次从黑箱开始猜：

## 7. General Lessons

以后遇到动画卡顿、跳动或闪烁问题，不能再把主观感觉当成事实输入，也不能再用切换 agent 或提高 reasoning 等级来替代诊断。更强的模型只能更好地利用输入；如果输入是错误的、主观的、缺少设备数据的，那么更强的模型也会更快地沿着错误假设推进。

开发者需要先问：

- 现象具体属于 FPS 低、draw 慢、Compose 重组、数据构建、位置输入不连续、smoother 重置，还是信息密度问题？
- 哪个指标证明这个判断？
- 当前改动能不能用一个字段验证？
- 是否可以先做最小 A/B？
- 改完后同一组指标是否改善？

推荐流程：

1. 先读真实代码路径，不要默认从最大文件开始改，也不要默认跳到 libs/native。
2. 如果系统还是黑箱，先加最小 debug-only trace。
3. 用 adb 从真实设备抓 10-15 秒数据。
4. 用脚本把 log 转成可读 Markdown。
5. 根据指标决定改 Canvas、Compose、window source、playback position smoothing，还是视觉密度。
6. 做窄改动。
7. 用同一组指标复测。

特别要注意：主观观测很重要，但它只负责描述现象。用户说“卡”“跳”“闪”是有效反馈，但不能直接推出“FPS 低”或“libs 数据太大”。这类判断必须由 debug 指标和设备数据支撑。

开发者也要承担提供有效输入的责任。对于需要设备交互的问题，agent 不一定能直接操作设备完成复现和 adb 捕获；即使能，成本也可能不合理。开发者应该主动把容易获得的真实信息提供给 agent：复现步骤、debug overlay 数值、adb log、格式化后的 `log.md`。这比反复切换模型或提高 reasoning 等级更有效，也更便宜。

这并不是 agent 做不到更深入的诊断，而是默认交互策略通常会优先选择低成本、局部、可执行的方案。大多数问题确实可以通过少量代码读取和局部修改解决；如果默认一上来就全量读代码、要求 adb、加 trace，会让小问题处理过重。过度诊断会增加时间、token 成本和用户等待，也会降低实际交互体验。

用户没有明确要求时，agent 倾向先用低成本路径推进。这符合实际产品体验：小问题快修，大问题再逐步升级。benchmark 或交互体验也会奖励更快、更少步骤的解决方式，而不是每次都做重型调查。所以更准确的结论是：agent 默认被优化为在信息不足时先尝试低成本、局部、可执行的方案；这对大多数任务是合理的，但遇到设备相关、动画相关、性能相关、长链路问题时，如果开发者不主动提供客观数据或要求建立观测，agent 就容易在低成本猜测路径里停留太久。

如果一个问题已经连续多轮没有解决，下一步不应该是继续猜，也不应该是继续调高思考等级，更不应该把“多买余额、换更强模型”当成解决方法，而应该强制升级为“数据获取”任务：加 trace、抓 log、做最小 A/B、确认哪个指标变坏。没有这个步骤，后续修改只是继续赌博。

这里的最小 A/B 不是“随便改一次看看”。它指的是：当存在两个或多个合理怀疑点，但还没有足够证据确认根因时，只选择一个变量做最小、可回退、可测量的改动，并用同一复现输入和同一组指标验证。

正确流程应该是：

1. 先写清假设，例如 A 是 Canvas primitive 太多，B 是 visual smoother 随 raw anchor 更新 reset。
2. 一次只改一个方向，例如只降低 primitive 数，或只调整 smoother key。
3. 使用同一段长音频、同一播放路径和同一组 adb/debug 指标复测。
4. 如果指标没有改善，就排除或降低这个方向的优先级，及时转向另一个假设。
5. 如果指标改善，再围绕这个方向继续收窄，而不是同时混入其他大改动。

最小 A/B 的价值在于控制变量。比如这次如果先只降低绘制 primitive，但 `drawAvgMs` 已经很低、`smoothReset` 和 `pxStepMax` 没变，就应该判断 Canvas 绘制不是主因；当 `smoothReset` 和 `rawUpdate/s` 对齐时，才应该转向 smoother reset。有效的 A/B 要让结果能回答“这个假设是不是主因”，而不是让多个改动混在一起后无法判断到底是谁起作用。

## 8. Follow-Up

当前卡顿和周期性跳动已经解决，但还有一个独立问题：Flash Visual 的 bit 信息密度可能偏高。现在性能正常、移动平滑后，人眼可能仍然跟不上过多 bit 动画。

这个后续应该作为可读性和视觉设计问题处理，而不是再混入性能问题。可能方向包括：

- 减少同屏可见 bit 数量。
- 缩短 visual viewport，让每个 bit 在画布上更宽。
- 加宽 bit 竖条或调整 ToneTracks / ToneEnergy 的视觉比例。
- 保持音频时间轴真实对齐，不为了可读性改变实际播放 timing。

这类调整应继续用用户观感和必要的 debug 数据配合判断，但不要再把“读不住”误判成“卡顿”。
