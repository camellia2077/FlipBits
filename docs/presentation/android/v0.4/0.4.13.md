## [v0.4.13] - 2026-05-15

### 新增功能 (Added)
* 新增 Android text follow character 消费链路，贯通 JNI、DTO、缓存恢复和播放映射，使播放器现在可以直接消费 libs 提供的 `token -> character -> byte` 结构化数据。这样 token 跟随不再只认可见字符，后续空格、换行和 mixed UTF-8 token 的播放位置都能按 libs 契约解析。
* 新增基于 structured character metadata 的 Binary/Hex divider 渲染能力：每个 byte 边界都会保留 divider，字符结束位置可以显示更强边界。这样 CJK 文本不再只是整块 UTF-8 byte 串，而能按字符内部 `8bits` 分组和字符边界显示。
* 新增 `PlaybackFollowDisplayMappingTest` 等跟随映射测试，专门覆盖 character kind、空白 character 显示和新的 divider 语义，避免 Android 侧再次回退到按 token 文本猜边界。

### 技术改进/重构 (Changed/Refactor)
* 重构 Android playback follow 数据模型，给 `PayloadFollowViewData` 增加 `textCharacters` 和新的 raw display unit character metadata，并同步更新 JNI bridge、`NativeAudioCodecGateway`、保存音频 decode cache 和输入分段聚合逻辑。这样实时生成、已保存音频和 UI 层都只消费同一套 libs follow contract。
* 重构 `PlaybackFollowDisplayMapping`、`PlaybackFollowTokenCard` 和 Binary/Hex annotation 逻辑，移除 Android 侧 `PayloadFollowDesignNormalizer` 这类“先合并/再猜字符”的兼容路径，改成直接根据 libs 的 character kind 与 byte metadata 决定 token 文本、空白占位和 divider 边界。
* 调整 Binary/Hex 注释排版策略：Binary 行保持每行最多 4 组，单个 `8bits` 组使用更稳定的固定占位；Hex 也统一收敛到每行最多 4 组，避免不同 token/字体条件下继续出现边界规则不一致。

### 修复 (Fixed)
* 修复 token 文本高亮只跟可见 characters 走、而 trailing space 对应的 Binary/Hex 仍在播放时 token 看起来已经结束的问题。Android 现在会按 libs 的 separator character 语义保留这些空白位置，播放到空格或换行所属 bytes 时，token 跟随与 Binary/Hex 跟随保持一致。
* 修复 CJK token 在 Binary/Hex 中缺少准确 divider 语义的问题。现在 Android 直接使用 libs 的 character-byte 映射来决定 byte divider 和字符边界，不再依赖 UI 层按文本内容手工推断。
* 修复大量测试 fixture 仍依赖旧 DTO 构造参数顺序的问题，删除 `TextFollowRawDisplayUnitViewData` 的兼容构造器，并把相关单测全部迁到新 DTO 形态，确保后续改动不会再悄悄回到旧 contract。
