## [v0.5.2] - 2026-05-24

### 新增功能 (Added)
* 新增 Android 生成进度的 operation snapshot / work-plan 消费模型，JNI 会把 `libs` 返回的 state、phase、overall progress、phase progress、work units、payload/segment 信息带到 Kotlin domain，Audio 页面进度条和状态文案可以直接使用同一份 native 生成进度数据。
* 新增 `EncodeProgressDisplayModel` 展示模型，把 Compose 页面需要的 percent、phase 和 busy 可见性从 `EncodeOperationSnapshot` 派生出来，避免页面参数链继续传递旧的 `encodeProgress / encodePhase` 字段名。

### 技术改进/重构 (Changed/Refactor)
* 重构 Android 生成流程，从旧的同步/callback/job progress 兜底路径切到 `create -> workPlan -> pump -> poll -> takeResult` 的 encode operation 生命周期；`NativeAudioCodecGateway` 现在连续推进 native work，并只按 UI 节奏节流 progress emission，避免把生成吞吐绑定到进度条刷新间隔。
* 调整生成结果边界，`EncodeAudioResult.Success` 现在只表示 PCM 已生成；raw/follow/lyrics/flash visual 数据继续在生成完成后通过 `buildEncodeFollowData(...)` 单独 hydration。这样播放可以先基于 PCM 进入成功路径，跟随数据不再被误认为 operation completion 的一部分。
* 重构 JNI follow hydration helper，区分 operation terminal PCM payload、encode follow hydration payload 和 decode payload，并补充字符串/数组 copy 的 buffer-size clamp、空 PCM buffer 保护、局部引用释放和 `ArrayList` 构造缓存，降低长文本 follow hydration 的 JNI 对象压力。
* 调整 Android state 和 debug 观察链路，`ModeAudioSessionState` 只保存 `encodeOperationSnapshot` / `encodeOperationWorkPlan` 作为生成进度事实来源；UI、debug scenario 和测试都改为从 snapshot 派生进度和 phase。
* 更新 Android 版本号为 `0.5.2`。

### 修复 (Fixed)
* 修复 Web 进度契约接入后 Android 生成可能卡在 `0%`、无法正常进入完成状态的问题。Android 现在直接消费 `libs` operation snapshot/work-plan，不再依赖旧 progress callback 或本地二次推导。
* 修复 Mini、Flash、Pro、Ultra 生成速度明显变慢的问题。Android operation loop 不再每 pump 一小段就固定等待 `33 ms`，而是在 worker dispatcher 上连续推进 native work、按阶段变化或约 33 ms UI cadence 发出进度；实机 capture 覆盖了 Mini/Flash/Pro/Ultra，避免速度回退。
* 修复 Flash 长文本生成在 native postprocess 阶段因重复复制完整 PCM 导致的额外拖慢。Flash long capture 现在可以稳定完成，并保持 follow hydration 后置。
* 修复旧 `encodeProgress / encodePhase` state 字段容易形成第二套进度事实来源的问题。兼容 getter 已从 state、测试和零散读点中移除，剩余同名参数只用于底层绘制数值输入。
