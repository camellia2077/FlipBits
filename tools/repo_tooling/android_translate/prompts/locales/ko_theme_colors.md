# Korean Theme And Color Translation Rules

Use this note when the Korean task covers theme titles, palette names, dual-tone descriptions, accessibility labels, or custom color import/export UI.

## Theme Titles

- Write theme titles as natural Korean product names.
- Avoid stiff literal transfer from English or Chinese when a smoother Korean title works.
- Keep titles compact enough for a settings picker.
- Prefer concise noun-phrase naming that reads naturally in Korean UI.

## Dual-Tone Descriptions

- Keep dual-tone descriptions short and color-led.
- Prefer compact color-list phrasing such as `색 A와 색 B.` or `색 A, 색 B, 색 C.`
- Do not turn the description line into atmospheric prose, lore copy, or marketing narration.
- Avoid repeating framing words like `듀얼 톤 팔레트` or `분위기` when the UI already provides the theme context.

## Accessibility Labels

- Prefer a clear pattern such as `이름 테마, 색 ...`
- Keep color names explicit and easy to scan in screen-reader output.
- Accessibility text should describe the visible palette directly, without decorative expansion.

## Material Palette Names

- Use natural Korean color names or stable loanwords when they read normally in a palette picker.
- Keep palette labels compact and product-like.
- Avoid mixing highly literary Korean wording with casual transliteration without a clear pattern.

## Product-Term Consistency

- Keep `듀얼 톤` as the stable product term when the UI is referring to the product/system name.
- Keep `Material` unchanged as the design-system family name.
